## 数据模型

使用下一迁移增加：

- `started_at`
- `completed_at`
- `abandoned_at`
- lifecycle idempotency 字段或独立事件表

建立最多一个 Active Session 的数据库保护：

```text
status in ('planned', 'in_progress')
```

如产品需要允许同时保留 learn/review 两个 Active Session，必须先更新 ADR；默认 V1 每用户只有一个 Active Session，减少离线和多设备冲突。

## API

建立版本化合同和路由：

```text
GET  /api/sessions/active
POST /api/sessions/:id/start
POST /api/sessions/:id/complete
POST /api/sessions/:id/abandon
```

所有 mutation：

- 用户从 Session 获得；
- 不接受 userId；
- 使用 idempotency key；
- 使用服务器时间；
- 在强制 RLS 事务中执行；
- 返回稳定机器码。

## 状态规则

- `start`：planned → in_progress；重复调用返回同一结果。
- `complete`：in_progress → completed；必须确认每个 Session Activity 至少有一条 accepted Attempt。
- `abandon`：planned/in_progress → abandoned；重复调用幂等。
- completed/abandoned 不能恢复为 active。
- completed/abandoned 不接受新 Attempt。
- Session 完成不代表每题正确，也不直接把内容标记掌握。

## Active Session

`GET active` 返回当前用户最新 active Session Header 和 Snapshot。没有则返回明确 `none`，不是 404 页面错误。

并发创建时：

- 数据库唯一约束是最终防线；
- API 捕获冲突并返回现有 Active Session；
- 不产生两个活跃计划。

## 测试

- 未认证拒绝；
- A/B 用户隔离；
- 合法状态转换；
- 所有非法逆向转换；
- 幂等 start/complete/abandon；
- 双设备并发；
- 未完成所有 Activity 时 complete 被拒；
- completed/abandoned 拒绝 Attempts；
- Active Session 恢复；
- 生命周期时间来自服务器；
- 迁移、RLS 和全仓回归。

## 验收

服务端可以唯一、确定、幂等地回答：

- 当前用户有没有未完成 Session；
- Session 是否已开始；
- 是否可以接受作答；
- 是否已完成或放弃。
